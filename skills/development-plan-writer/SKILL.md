# 程式開發計畫撰寫

## 目的

使用此 Skill 將技術選型、ADR、需求描述、資料流或既有規劃內容，整理成兩段式交付文件：

1. `開發計畫框架`：給開發者看的實作藍圖，包含資料邊界、模組拆分、處理流程、技術盲點與驗收方向。
2. `PM 確認流程`：給 PM 或非實作者確認的簡化流程，聚焦程式目標、資料來源、整體流程、寫回結果、暫存成果與待確認事項。

## 模式選擇

| 使用者要求 | 使用模式 | 讀取 reference |
|---|---|---|
| `開發計畫框架`、`程式開發計畫`、`實作計畫`、`開發藍圖`、`模組規劃` | 開發計畫模式 | `references/development-plan-template.md` |
| `PM確認流程`、`PM 確認版`、`簡易流程`、`流程確認文件`、`給 PM 看`、`手動執行流程` | PM 流程模式 | `references/pm-flow-template.md` |
| `根據這份開發計畫產出 PM 流程`、`把開發計畫轉 PM 版` | 轉換模式 | 先讀 `references/pm-flow-template.md`，並以使用者提供的開發計畫為唯一主要來源 |

## 使用流程

1. 先判斷使用者要的是開發計畫、PM 流程，還是由開發計畫轉 PM 流程。
2. 若使用者要求開發計畫，從技術選型、ADR、需求描述與資料流中萃取實作資訊，使用開發計畫模板。
3. 若使用者要求 PM 流程，優先使用已確認的開發計畫作為來源；不要新增未被確認的技術假設。
4. 若資訊不足但可合理推斷，標示「需依實際設定確認」或「待確認」。
5. 若缺少資訊會改變流程範圍或寫回責任，最多提出 3 個釐清問題；否則直接產出草案。

## 撰寫規則

- 一律使用繁體中文。
- 技術名稱、檔名、設定鍵、欄位名稱、API 參數、資料表名稱保留原文並使用 inline code。
- 開發計畫模式可以保留工程細節，例如 `Console CoreHost`、`SDE`、`weather.gdb`、`appsettings.json`、`NLog`、spatial join、API 條件與模組名稱。
- PM 流程模式要避免過深實作細節，重點放在 PM 需要確認的資料來源、流程、寫回欄位、執行方式與範圍。
- 不要在文件中寫實作程式碼；若需要流程，使用表格或 `text` 流程圖。
- 若文件涉及 SDE/GDB/FeatureClass/API/MSSQL/排程，必須明確說明資料來源、寫回目標、暫存成果與維護風險。

## 品質檢查

輸出前確認：

- 是否明確說明程式目標與不負責範圍。
- 是否交代資料來源、輸入條件、輸出結果與寫回位置。
- 是否區分來源資料、暫存成果與最終寫回資料。
- 是否列出待確認事項與驗收方向。
- PM 流程是否足夠簡潔，且沒有引入開發計畫中未確認的新假設。
