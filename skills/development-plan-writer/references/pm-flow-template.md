# PM 確認流程模板

當使用者要產出「PM 確認流程」、「簡易流程」、「給 PM 看」、「流程確認文件」或要把已確認的開發計畫轉成 PM 版時，使用此模板。

## 文件開頭

```md
# {功能或程式名稱}流程
```

可視使用者需求保留簡單 HTML/CSS 樣式，但預設輸出純 Markdown。

## 建議章節

### 1. 程式目標

用 1 到 2 段說明此程式要完成的事情、何時執行、由誰執行，以及不做什麼。

建議用表格摘要：

| 項目 | 說明 |
|---|---|
| 執行方式 |  |
| 主要篩選條件 |  |
| 主要寫回欄位 |  |
| 主要結果 |  |

### 2. 來源檔案 / 資料來源

列出 PM 需要理解的資料來源，不需要展開過深的類別或方法細節。

| 類型 | 來源 | 說明 |
|---|---|---|

若有 API 查詢條件，可用表格簡化：

| 用途 | API 查詢條件 | 回傳資料用途 |
|---|---|---|

### 3. 整體流程

用 `text` 流程圖或編號步驟說明主要流程。

```text
1. 執行前設定
   - ...

2. 篩選待處理資料
   - ...

3. 取得資料並比對
   - ...

4. 組合寫回內容
   - ...

5. 寫回結果
   - ...
```

### 4. 寫回結果

說明最終會更新哪個欄位、寫入什麼類型的內容，以及是否有長度、格式或覆寫規則。

| 類型 | 寫回內容 |
|---|---|

### 5. 暫存成果 / 中間成果

若流程會產生 GDB、暫存表、log、報表或中間檔案，需列給 PM 確認。

| 暫存成果 | 用途 |
|---|---|

### 6. PM 需確認事項

列出 PM 或需求方需要確認的範圍、資料來源、執行方式、寫回欄位與不做事項。

| 確認項目 | 說明 |
|---|---|

## 轉換規則

從開發計畫轉 PM 流程時：

- 保留程式目標、資料來源、執行方式、主要流程、寫回結果與暫存成果。
- 移除或壓縮模組設計、Phase、class/function、錯誤矩陣等工程細節。
- 不新增開發計畫沒有提到的資料來源、API 條件或寫回欄位。
- 將「技術盲點」改寫成「PM 需確認事項」。
- 若開發計畫中標示待確認，PM 版也必須保留待確認狀態。
